## Conventions worth knowing

### Git remote
The clone's remote URL still says `git@github.com:MaastrichtU-IDS/pymos.git`
because the local checkout pre-dates the rename. GitHub auto-redirects, so it
works. If you want to clean it up:

```bash
git remote set-url origin git@github.com:MaastrichtU-IDS/omny.git
```

### Branches
- `master` is the only long-lived branch.
- Feature branches: `perf/...`, `fix/...`, `chore/...`, `docs/...`, `release/...`
- Squash-merge via PR; CI must pass on all 3 jobs.
- A sibling worktree at `/data/dumontier/pymos-run` may have `master` checked out
  by another session — **do not pull or commit there from this session**.

### Local proxy + SSH
Host is behind `proxy.unimaas.nl:3128`. SSH to GitHub goes through
`nc CONNECT proxy.unimaas.nl:3128`. Already configured in `~/.ssh/config`;
nothing to do.

### Rootless docker
Socket: `/run/user/1007/docker.sock`. Data root: `/data/dumontier/docker`.
Proxy env wired. Just `docker ...` works.

## Release flow (canonical, do not deviate)

```bash
# 1. Bump version + CHANGELOG in one commit on master
sed -i 's/^version = .*/version = "0.X.Y"/' pyproject.toml
# edit CHANGELOG.md — add a ## [0.X.Y] — YYYY-MM-DD section at the top
git commit -am "release: 0.X.Y"
git push

# 2. Tag (must match pyproject.toml exactly) and push
git tag -a v0.X.Y -m "release 0.X.Y"
git push origin v0.X.Y
```

`publish.yml` runs in ~40s. The **tag-vs-pyproject guard** (added in PR #60)
fails the build step with a clear error if `pyproject.toml` doesn't match
the tag. **History to know:** `v0.2.0` and `v0.1.3` were tagged without
bumping pyproject; both publish runs silently uploaded `omny-0.1.1.*` which
PyPI rejected as duplicates. PyPI doesn't reflect those tags — only 0.1.0,
0.1.1, 0.2.1, 0.2.2 are live. **Don't try to rescue v0.2.0 or v0.1.3** —
just leave their failed runs as historical record.

## Notebook server (Docker Compose)

From `/data/dumontier/pymos/examples/`:

```bash
docker compose up --build -d
# JupyterLab:   http://localhost:8888  (no token, demo config)
# Oxigraph:     http://localhost:7878
docker compose down  # when done
```

The compose mount is `..:/workspace`, so the container sees the live
working tree. The pre-installed deps (PR #63) cover all 6 notebooks
including 06 (owlrl + HermiT). Notebook 01–04 + 06 verified to run
end-to-end as scripts inside the container; 05 needs the JupyterLab
UI for the `%%mos` magics.

**Watch out:** if the running container was built from an old checkout
(this happens because `/data/dumontier/pymos-run` is another session's
worktree and `docker compose` started there at some point), the
container's `pip install -e .` may point at the old package name.
Symptom: `import omny` fails inside the container. Fix: tear down +
re-up from `/data/dumontier/pymos/examples/`:

```bash
docker stop examples-notebook-1 examples-triplestore-1
docker rm  examples-notebook-1 examples-triplestore-1
cd /data/dumontier/pymos/examples
docker compose up --build -d
```

   `locstr` correctly. Limited scope; tests would need to assert
   round-trip equality through both lang-tagged + typed literals.
